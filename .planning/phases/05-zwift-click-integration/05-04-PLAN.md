  <how-to-verify>
    Pre-flight (no hardware needed yet):
    1. `cd engine && uv run pytest tests/ -q` — confirm all tests are green before unplugging the laptop. Expected: exit 0.
    2. Verify `engine/engine/input/click.py` constants match `docs/click-ble-spike.md` "Confirmed values" column. If they diverge, stop and fix in code.

    Setup the rig:
    3. KICKR Core powered on; bike attached.
    4. Zwift Click freshly powered (replace battery if it has not been used recently).
    5. macOS Bluetooth permission granted to the terminal (System Preferences → Privacy & Security → Bluetooth).
    6. Cockpit UI dev server running (`cd cockpit && npm run dev` or whatever Phase 3 used) and opened in a browser.

    **Test 1 — Engine boots without Click present (no-Click resilience).**
    7. Power OFF the Click (or move it out of BLE range).
    8. `cd engine && uv run python -m engine`.
    9. Open the cockpit. Pedal briefly. Confirm:
       - Telemetry flows (speed/power/cadence visible in cockpit, just like Phase 4).
       - Keyboard k/j still shifts the gear (Phase 2 fallback).
       - Engine log shows a warning like "Zwift Click not found; retrying" but NO traceback, NO crash.
       - No `click_status` message with `connected: true` appears.
    10. Ctrl-C. Engine shuts down cleanly within 15 s.

    **Test 2 — Engine connects to Click on startup.**
    11. Power ON the Click. Wait 3 seconds for it to start advertising.
    12. `cd engine && uv run python -m engine`.
    13. Within ~30 s, engine log shows "Zwift Click connected and notifying".
    14. Cockpit (if it consumes click_status) shows the indicator. If the cockpit does not yet render an indicator, that is acceptable for this phase — the contract is that the WS message is sent.
    15. Verify with the browser DevTools WS frame inspector: a frame with `{"type":"click_status","connected":true}` was received.

    **Test 3 — Plus button shifts up.**
    16. Note the current gear shown in the cockpit.
    17. Press Click PLUS once. Within ~250 ms, the cockpit gear number increments by 1.
    18. Repeat 4 more times slowly (>200 ms apart). Each press = one gear up. Verify gear cap at 10 — pressing plus at gear 10 does not roll over or crash.

    **Test 4 — Minus button shifts down.**
    19. Press Click MINUS once. Cockpit gear decrements by 1.
    20. Repeat 4 more times slowly. Verify gear floor at 1 — pressing minus at gear 1 does not roll under or crash.

    **Test 5 — Debounce on rapid press.**
    21. From gear 5, press PLUS 5 times as fast as physically possible (within ~300 ms total).
    22. The gear must NOT have advanced by 5 — it should advance by ~3 or fewer (one shift per 100 ms window). Note the actual count in the verification doc.
    23. From the resulting gear, press MINUS 5 times rapidly. Same debounce behavior expected on the down direction.

    **Test 6 — Keyboard fallback still works while Click is connected.**
    24. With Click still connected, press `k` in the engine terminal. Cockpit gear increments.
    25. Press `j`. Cockpit gear decrements.
    26. Both shifters work simultaneously — pressing `k` then PLUS then `k` produces three shift-ups.

    **Test 7 — Click disconnect mid-ride.**
    27. While the engine is running, power OFF the Click (or remove its battery).
    28. Within ~5 s, engine log shows a disconnect warning. Cockpit receives `{"type":"click_status","connected":false}`.
    29. Engine does NOT crash. Keyboard still shifts.
    30. Power Click back ON. Engine should reconnect (may take 30-60 s for next scan window). New `click_status: connected:true` arrives. Plus/minus buttons resume shifting.

    **Test 8 — Clean shutdown.**
    31. Ctrl-C the engine. Shutdown completes within 15 s. No "task was destroyed but it is pending" warnings in the log. KICKR receives FTMS Stop + Reset (Phase 2 INFRA-02 unchanged).

    For each test, record PASS / FAIL / SKIPPED in the verification doc. Tests 1, 6, and 8 are the critical "fallback never broke" gates; if any of those FAIL, do not sign off — file gap-closure work via `/gsd:plan-phase --gaps` instead.

    Resume signal: "phase 05 verified" (with PASS/FAIL counts), OR "phase 05 deferred: <reason>" if hardware is not currently available — STATE.md and ROADMAP.md will record the partial state.
