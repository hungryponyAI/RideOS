  <how-to-verify>
    Step-by-step instructions for the user (Claude pre-populates the doc with these as a checklist):

    1. **Power on the Zwift Click** (insert/replace battery if it has not been used recently). It should advertise as soon as a button is pressed.
    2. **Open nRF Connect** on a smartphone (iOS or Android). Start a BLE scan.
    3. Press a button on the Click. A new device should appear, typically named "Zwift Click" but possibly nameless on first scan.
    4. **Record advertising data** in the doc (under section "Advertising packet"):
       - Local name (or "no name" if absent)
       - Manufacturer data: hex bytes — expected first 2 bytes little-endian = `4A 09` (manufacturer ID `0x094A`), expected byte index 2 (3rd byte) = `0x09` (Click device type). Record the actual raw bytes seen.
       - Service UUIDs in advertisement — expected EITHER `00000001-19ca-4651-86e5-fa29dcdd09d1` (long form) OR `FC82` (16-bit short form, Jan 2025 firmware). Record what is actually advertised.
    5. **Connect** to the device in nRF Connect. Browse the GATT services.
    6. **Record the custom Zwift service** in the doc (under section "GATT services"):
       - Service UUID (full UUID as shown by nRF Connect)
       - List ALL characteristic UUIDs under it with their properties (READ / WRITE / WRITE-NO-RESPONSE / NOTIFY / INDICATE)
       - Identify which characteristic has NOTIFY → that is ASYNC. Expected: `00000002-19ca-4651-86e5-fa29dcdd09d1`.
       - Identify which characteristic has WRITE / WRITE-NO-RESPONSE → that is SYNC_RX. Expected: `00000003-19ca-4651-86e5-fa29dcdd09d1`.
    7. **Enable notifications** on the ASYNC characteristic. Press a button — bytes should appear in the notification log.
    8. **Try the unencrypted handshake** (LOW confidence — may not work on current firmware):
       - On the SYNC_RX characteristic (the WRITE one), use nRF Connect's "Send" feature to write the 6-byte payload `52 69 64 65 4F 6E` (ASCII "RideOn") with response=false / write-no-response.
       - After writing, press plus and minus and watch the ASYNC notifications.
       - **If you see frames starting with byte `0x37`** containing button data: unencrypted path WORKS — record this.
       - **If only `0x15` idle frames or no useful data appears after pressing buttons**: unencrypted path fails — record this; Plan 05-02 will need ECDH fallback.
    9. **Capture button bytes** (regardless of whether handshake succeeded — even encrypted firmware may emit some plaintext metadata):
       - Press PLUS button five times slowly. Record the raw hex byte string of each notification frame in section "Plus button captures".
       - Press MINUS button five times slowly. Record the raw hex byte string of each notification frame in section "Minus button captures".
       - Press PLUS and HOLD. Record the press frame and the release frame separately if both appear.
       - Capture 10 seconds of idle (no button press) into section "Idle frames" — these are the heartbeat frames the implementation must IGNORE.
    10. **Decode the bytes** and fill in the "Confirmed values" table at the top of the doc:
        - First byte of plus-press frame (expected `0x37`)
        - First byte of minus-press frame (expected `0x37`)
        - Where in the payload the value `0` (pressed) vs `1` (released) appears
        - Which "key" (field tag) is plus and which is minus (research expects key '1' = plus, key '2' = minus)
    11. **Document the firmware version** if visible: in nRF Connect, look under Device Information service → Firmware Revision String. Record the version.
    12. Save the file. Do NOT alter any code yet — this plan only produces documentation.

    Resume signal: "spike done" plus a paste of the "Confirmed values" table from the doc, OR "spike failed: <reason>" if the Click cannot be brought up at all (in which case the phase pauses until hardware is available).
